# 科研任务的 AI 协作流程设计说明

## 1. 文档目标

本文档说明：当用户输入一个科研主题后，AI 系统应如何以结构化流程完成以下任务：

- 拆解研究问题
- 检索并整理引文
- 生成方法论方案
- 组织论证结构
- 通过评估机制持续优化输出
- 在关键节点接收用户反馈并纠偏
- 最终合成为可审阅的科研草案

这类任务不适合单轮一次性生成，更适合采用“**任务分治 + 评估迭代 + 人类反馈**”的组合流程。LangGraph 官方将常见流程分为 `parallelization`、`routing`、`orchestrator-worker`、`evaluator-optimizer` 等模式；其中，多子任务研究最接近 `orchestrator-worker`，而质量打磨最接近 `evaluator-optimizer`。 [oai_citation:0‡LangChain 文档](https://docs.langchain.com/oss/python/langgraph/workflows-agents?utm_source=chatgpt.com)

---

## 2. 为什么科研任务不能只靠一次生成

科研类任务通常同时包含多个不同性质的子工作：

- 研究主题拆解
- 文献检索与筛选
- 方法论设计
- 实验设计
- 论证链搭建
- 局限性分析
- 用户偏好对齐

这些工作既有并行性，也有强依赖关系，因此不适合简单的“用户提问 → 模型一次回答”模式。LangChain 官方文档指出，复杂任务往往需要把流程拆成离散节点，并通过状态和条件边进行编排；LangGraph 本身就是为持久化执行、状态流转、调试和 human-in-the-loop 这类场景设计的。 [oai_citation:1‡LangChain 文档](https://docs.langchain.com/oss/python/langgraph/workflows-agents?utm_source=chatgpt.com)

从工程视角看，科研辅助系统最合理的思路不是让一个模型一次性完成所有事情，而是：

1. **先把任务拆开**
2. **让不同模块分别处理**
3. **汇总成阶段性草案**
4. **对草案进行结构化评估**
5. **针对低分部分定向优化**
6. **在关键节点让用户介入纠偏**

## 3. 适合科研任务的总体流程

## 3.1 推荐总架构

科研场景最适合的不是单一 workflow，而是两层结构：

- **任务分治层：Orchestrator-Worker / Hierarchical**
- **质量闭环层：Evaluator-Optimizer**
- **交互控制层：Human Feedback Gate**

也就是说，系统应采用：

**“先分治研究，再迭代优化，再由用户在关键节点纠偏”**

其中：

- `orchestrator-worker` 负责拆解任务、分发子任务、汇总结果。LangChain 官方对该模式的描述就是：将任务拆分为子任务、委派给 worker、最后综合 worker 的输出。 [oai_citation:2‡LangChain 文档](https://docs.langchain.com/oss/python/langgraph/workflows-agents?utm_source=chatgpt.com)
- `evaluator-optimizer` 负责生成—评估—修订闭环。Anthropic Cookbook 将其定义为：一个模型负责生成，另一个模型负责评估并给出反馈，若不达标则循环优化。 [oai_citation:3‡Claude平台](https://platform.claude.com/cookbook/patterns-agents-evaluator-optimizer?utm_source=chatgpt.com)
- `human-in-the-loop` 负责让用户在关键节点直接参与评估或重定方向；LangGraph 官方明确将 human-in-the-loop 视为核心能力之一。 [oai_citation:4‡LangChain 文档](https://docs.langchain.com/oss/javascript/langgraph/overview?utm_source=chatgpt.com)

## 4. 科研任务的标准流程

## 4.1 第一步：接收研究主题与约束条件

用户输入不仅应包含“科研主题”，还应尽量包含这些约束信息：

- 研究目标
- 学科方向
- 时间范围
- 是否偏综述 / 偏实验
- 需要中文还是英文文献
- 引文风格要求
- 输出形式（提纲、研究方案、方法论草案、论文框架等）

这一阶段的目标不是直接回答，而是形成一个**清晰的任务定义**。

---

## 4.2 第二步：Planner 拆解研究任务

系统首先进入 **Planner** 阶段，将科研主题拆成多个子问题。  
LangGraph 官方建议，在构建 agent 时，先将流程映射为离散步骤，再为每个步骤定义职责和状态。 [oai_citation:5‡LangChain 文档](https://docs.langchain.com/oss/python/langgraph/thinking-in-langgraph?utm_source=chatgpt.com)

对于科研任务，Planner 一般需要拆出以下模块：

- 研究背景与问题定义
- 相关工作 / 引文检索
- 方法论候选方案
- 实验设计
- 局限性与风险
- 预期贡献与创新点

这一步的关键不是写正文，而是明确研究问题边界，形成一个可执行的研究计划。

---

## 4.3 第三步：Orchestrator 将任务分发给多个 Worker

进入 **Orchestrator / Router** 阶段后，系统把子任务分配给不同 worker。  
这一步最接近 `orchestrator-worker` 模式：先拆任务，再将子任务交给专门模块处理，最后统一汇总。 [oai_citation:6‡LangChain 文档](https://docs.langchain.com/oss/python/langgraph/workflows-agents?utm_source=chatgpt.com)

推荐至少包含以下 worker：

### Citation Worker
负责：
- 检索候选文献
- 提取摘要与关键信息
- 判断相关性和时效性
- 区分经典文献与近年文献

### Method Worker
负责：
- 生成方法论候选
- 明确变量、数据、流程、指标
- 判断方法与研究问题是否匹配
- 输出实验设计草案

### Argument Worker
负责：
- 组织论证结构
- 明确研究背景、研究空白、方法路径、预期贡献
- 保证逻辑链条完整

对于复杂研究问题，这种“按职责拆 worker”的方式比单模型整篇生成更可控，也更利于后续定向优化。LangChain 也提醒，多代理或多模块适合复杂工作流，但应围绕明确分工来设计，而不是无序堆叠复杂性。 [oai_citation:7‡LangChain 文档](https://docs.langchain.com/oss/python/langchain/multi-agent?utm_source=chatgpt.com)

---

## 4.4 第四步：Draft Aggregator 聚合阶段性草案

当各 worker 完成初稿后，系统进入 **Draft Aggregator** 阶段，将不同模块合成为一份阶段性研究草案。

此时输出的内容通常包括：

- 研究背景
- 文献综述 / 核心引文
- 方法论草案
- 实验设计建议
- 预期贡献
- 局限性说明

这一阶段的目标不是最终定稿，而是形成一版**可评估的中间稿**。

---

## 4.5 第五步：Evaluator 对草案进行结构化评估

这是科研流程中最关键的一步。

`evaluator-optimizer` 模式适合那些“有明确成功标准，但需要多轮迭代才能达标”的任务；科研写作、方法论设计、文献组织都非常符合这个特点。 [oai_citation:8‡LangChain 文档](https://docs.langchain.com/oss/python/langgraph/workflows-agents?utm_source=chatgpt.com)

Evaluator 不应只输出一句“还可以优化”，而应输出**结构化评估结果**，至少包括：

- 各维度分数
- 优点
- 缺点
- 修改建议
- 是否通过
- 需要修订的模块

推荐的评估维度如下：

### 1. 引文真实性与可追溯性
检查文献是否真实存在、信息是否完整、是否能支撑对应论点。

### 2. 引文相关性与覆盖度
检查文献是否真正贴合主题，是否覆盖关键研究方向。

### 3. 引文时效性
检查近 3–5 年研究是否充分覆盖，是否只有旧文献没有新进展。

### 4. 方法论严谨性
检查研究问题是否被操作化，方法步骤是否清晰，变量和指标是否明确。

### 5. 实验设计可执行性
检查数据来源、实验流程、评价指标是否可落地。

### 6. 论证链完整性
检查背景、问题、研究空白、方法、贡献之间是否逻辑闭环。

### 7. 用户意图对齐度
检查输出是否真正符合用户要求，例如偏综述、偏实证、偏工程、偏近年英文文献等。

科研系统里，“用户意图对齐度”是一个非常重要但常被忽略的评估项。技术上写得对，不等于用户真正需要。

---

## 4.6 第六步：Human Feedback Gate 在关键节点接收用户反馈

科研辅助流程不应完全自动闭环，而应在关键节点提供 **human-in-the-loop**。LangGraph 官方将 human-in-the-loop 作为一类核心能力，适用于需要人工确认、纠偏或决策的流程。 [oai_citation:9‡LangChain 文档](https://docs.langchain.com/oss/javascript/langgraph/overview?utm_source=chatgpt.com)

建议在以下三个 checkpoint 暂停并允许用户输入反馈：

### Checkpoint A：研究计划完成后
用户可以确认：
- 研究范围是否准确
- 子问题是否完整
- 方向是否跑偏

### Checkpoint B：引文与方法草案完成后
用户可以确认：
- 文献是否足够新
- 方法是否偏理论或偏实证
- 是否需要更强调工程可落地性

### Checkpoint C：最终合成前
用户可以根据 evaluator 报告决定：
- 是否继续优化
- 是否改方向
- 是否接受当前版本

这一层的作用是防止系统“技术上正确，但方向上偏离”。

---

## 4.7 第七步：Optimizer 只重写低分模块

如果 Evaluator 判定未通过，系统不应整篇全文重写，而应进入 **Optimizer**，只对低分模块进行定向修订。

这是工程稳定性的关键做法。

原因有三点：

1. 全文重写容易把已经正确的部分写坏
2. 科研文本通常模块性强，更适合局部返工
3. 局部优化比全局重生成更节省 token，也更容易追踪修改原因

Optimizer 的输入应包括：

- evaluator 的结构化报告
- revision targets（需修订模块）
- 用户新增反馈
- 当前合格模块的锁定内容

这样可以让系统做到“保留正确部分，只修错误部分”。

---

## 4.8 第八步：循环评估，直到通过或回到 Planner 重规划

修订完成后，系统应重新进入 Evaluator。  
形成闭环：

**生成 → 评估 → 反馈 → 优化 → 再评估**

如果只是质量问题，则循环在 Evaluator 与 Optimizer 之间即可。  
如果用户反馈属于方向级变更，例如：

- 从综述改成实验设计
- 从教育场景改成医疗场景
- 从定性研究改成定量研究

那么系统不应只做局部优化，而应回到 Planner 重新规划任务。  
这类“replan”能力非常重要，否则系统会在错误任务定义上越修越偏。

---

## 4.9 第九步：Synthesizer 合成最终科研输出

当各评估维度达到阈值后，系统进入 **Synthesizer** 阶段，将所有通过评估的模块合成为最终输出。

最终成果可以是：

- 研究方案草稿
- 论文提纲
- 方法论设计书
- 相关工作综述
- 研究路线图
- 带引文的结构化初稿

这一步的目标不是再创造新内容，而是做最终整合、术语统一、格式收束和逻辑连贯性检查。

---

## 5. 推荐的流程图

下面是一版推荐流程：

```text
User Input
  ↓
Planner
  ↓
Orchestrator / Router
  ├─ Citation Worker
  ├─ Method Worker
  └─ Argument Worker
  ↓
Draft Aggregator
  ↓
Evaluator
  ├─ pass → Synthesizer → Final Output
  └─ fail → Human Feedback Gate / Optimizer
                   ↓
               Optimizer
                   ↓
               Evaluator
